# 기획서 → 테스트케이스 → 테스트 코드 자동화 (QA 워크플로 개선)

모바일 앱 및 서비스(에이닷, SK텔레콤) QA를 진행하며, **테스트 설계·작성에서 반복 비중이 가장 큰 두 단계를 자동화**해 QA 팀의 시간을 실제 테스트·품질 검증에 더 쓸 수 있도록 만든 프로젝트입니다.

1. 기획서(PDF) → 테스트케이스(TC) 작성 *(실사용 단계)*
2. 테스트케이스 → Appium(Python) 자동화 코드 변환 *(고도화 진행 중)*

**QA 업무를 하며 직접 병목을 발견하고 개인적으로 설계·구현해 팀 업무 흐름에 도입한 프로젝트입니다.**

> 사내 도구를 익명화한 포트폴리오 버전입니다. 실제 데이터·기획서·로케이터·모델 가중치는 제외했습니다.

---

## 풀고자 한 문제 (QA 관점)

- TC 작성과 자동화 코드 변환은 반복적이지만, 화면 구조·로케이터·예외 케이스 같은 **도메인 맥락**이 필요해 단순 템플릿으로는 한계가 있었습니다.
- 작업량이 많다 보니 작성자·시점에 따라 **TC와 코드 스타일이 들쭉날쭉**했고, 이는 테스트 유지보수 비용으로 이어졌습니다.
- 사내 기획서를 외부 API로 보낼 수 없는 **보안 제약**이 있었습니다.

→ "반복 작업을 줄이되, **품질은 사람(QA)이 끝까지 책임지는** 구조"를 목표로 로컬 자동화 파이프라인을 설계했습니다.

---

## 고안하여 도입한 것

### 1. QA가 품질 게이트키퍼인 사람-검수 루프
모델이 코드 초안을 생성하고 → **QA가 검수·트림·승인**한 코드만 실제 테스트셋/학습셋에 반영하는 구조를 만들었습니다.
- 검증 불가능한 케이스(웹뷰, 만들 수 없는 사전조건 등)는 추측 코드로 가짜 통과를 만들지 않고 **사유와 함께 SKIP**으로 명시 추적
- 승인 시 트림한 분량을 기록 → "과하게 생성된 스텝"을 줄이는 기준으로 활용
- 모델이 좋아질수록 초안 품질이 올라가 **리뷰 부담이 줄어드는 선순환**

### 2. "신뢰할 수 있는 테스트"를 위한 장치
자동 생성 테스트의 가장 큰 리스크는 **그럴듯해 보이지만 실제로는 검증하지 않는 코드**입니다. 이를 막기 위해:
- **로케이터 grounding** — 실존하는 로케이터 목록을 모델에 함께 제공해, 존재하지 않는 셀렉터를 지어내 테스트가 깨지는 것을 차단
- **품질 게이트** — 문법 검증(`ast.parse`), 반복/과생성 탐지, 타깃 대비 커버리지 측정으로 "도는 코드 + 맞는 검증"인지 확인
- 측정 가능한 지표를 도입해, 감이 아니라 수치로 품질을 판단

### 3. 검수에서 잡아낸 실제 결함 패턴 (자동 생성 코드 리뷰)
QA 도메인 지식으로 직접 걸러낸, 자동 생성 테스트에서 반복되던 결함들:
- `assert el.click()` → `click()`은 None을 반환해 **항상 실패**
- `el.is_displayed()` 단독 호출 → 아무것도 검증 못 하는 **no-op** (assert 필요)
- 존재하지 않는/오타 로케이터 → 런타임 에러
- 화면 전환 후 이전 요소 재사용 → **stale element** 로 깨짐
- `except: pass` → 실패를 삼켜 **깨져도 통과하는** silent failure (가장 위험)
- 비활성 버튼을 `element_to_be_clickable`로 대기 → 타임아웃 (상태 검증엔 `presence_of_element_located`)
- TC 스텝 수 대비 **과생성된 스텝 트림**

이 패턴들을 문서화해 검수 기준으로 삼았고, 자동 검사(미정의 로케이터/헬퍼 탐지)로 일부를 자동화했습니다.

### 4. 지표 게이밍 경계 (정성 검증의 중요성)
생성 다양성을 높이는 기법(best-of-N)을 적용했더니 커버리지 지표가 78% 증가해 개선처럼 보였으나,
**직접 읽어보니 같은 내용을 표현만 바꾼 가짜 커버리지**였습니다. 자동 지표만 믿지 않고 정성 검증을 병행해야 한다는 점을 확인한 사례입니다.

### 5. Slack에서 테스트 실행 (CI 자동화)
팀원 누구나 **Slack 슬래시 커맨드 한 줄로** 자동화 테스트를 실행하고 결과를 Slack에서 받습니다. CI 콘솔에 들어가지 않아도 돼서 테스트 실행 진입장벽을 낮추고 팀 전체의 접근성·가시성을 높였습니다.
- `/test start android all 36042` 처럼 플랫폼·디바이스·앱환경·TC를 한 명령으로 지정
- Slack 슬래시 커맨드 → (Flask 서버) GitHub Actions `workflow_dispatch` 트리거 → self-hosted 러너에서 Appium 실행 → 결과 Zephyr 동기화 + Slack 회신
- Slack 서명(HMAC) 검증·허용 사용자 제한으로 보안 처리
- 코드: [`ci/`](ci/) — 테스트 **생성에서 실행·리포팅까지** 자동화로 연결

---

## 성과

- TC 생성(실사용 단계): 페이지 파싱 성공률 **148/149 (99.3%)**, 비정상 반복 생성 **0/149**
- 코드 생성(고도화 진행 중): 현재 문법 유효 **31/39 (79%)**, 검수 게이트를 통과한 코드만 반영. 데이터 확충·품질 개선 진행 중
- 반복적인 TC·코드 작성 시간을 줄이고, 생성물의 **스타일 일관성**을 확보
- 데이터 규모보다 **정제 품질**이 결과를 좌우함을 정량적으로 확인 (잘 정제한 약 150건으로 실사용 품질 도달)

## 왜 이렇게 만들었나 (기술 선택 근거)

도구는 "최신이라서"가 아니라 **QA 현장의 제약 조건**에 맞춰 골랐습니다.

- **왜 LLM인가** — TC·코드 변환은 기획서의 자연어를 이해하고 화면 구조·예외 케이스를 반영해야 해서, 규칙/템플릿 방식으로는 다양한 기획서를 일반화하기 어려웠습니다. 자연어 → 구조화 매핑은 LLM이 적합했습니다.
- **왜 로컬(클라우드 API 대신)인가** — 사내 기획서를 외부 API로 보낼 수 없는 보안 제약이 핵심 요건이었습니다. Apple Silicon에서 도는 MLX 기반으로 구성해 **데이터가 기기 밖으로 나가지 않고, 호출 비용도 0**입니다.
- **왜 프롬프팅이 아니라 파인튜닝인가** — 산출물이 사내 포맷(step_reporter, elements.py 로케이터, Zephyr TC JSON)과 관용구를 정확히 따라야 했는데, 범용 모델 프롬프팅만으로는 일관성이 떨어졌습니다. 검수한 페어로 파인튜닝해 **포맷·스타일을 안정화**했습니다.
- **왜 이 모델들인가 (제약에 맞춘 선택)**
  - TC 생성: **EXAONE-3.5 7.8B** — 기획서·TC가 한국어라 한국어 특화 모델이 유리
  - 코드 생성: **Qwen2.5-Coder 7B** — 코드 특화 모델이 동급 범용 모델보다 Python/Appium 생성 품질이 높음
  - OCR: **Qwen2.5-VL 7B** — 기획서에 이미지·표·스캔이 섞여 있어, 텍스트 추출이 안 되는 페이지를 VLM으로 보완
  - 공통: **7B·4bit** — 보유한 맥(16~32GB)에서 로컬 학습·추론이 가능한 크기. 더 큰 모델은 로컬 실행이 어려움

### 구성
- **두 트랙** — `finetune/`(기획서 PDF → TC), `code_gen/`(TC → Appium 코드)
- **하이브리드 OCR** — 텍스트 PDF는 pymupdf, 스캔 페이지만 VLM으로 처리해 기획서 내용 누락 최소화
- **Jira/Zephyr 연동** — TC 조회 및 결과 업로드 (`jira_client.py`)
- **GUI** — 기획서 업로드 → TC 생성 → 업로드까지 한 화면에서 (Gradio)
- **Slack → CI** — Slack 슬래시 커맨드로 GitHub Actions 테스트 실행·결과 회신 (`ci/`)

학습=추론 프롬프트 일치, 평가지표 정정 등 세부 고민은 코드 주석과 `code_gen/`·`finetune/` 스크립트에 정리되어 있습니다.

## 진행 상황과 확장 방향

- **기획서 → TC 생성**: 실사용 가능한 수준 (파싱 99.3%, 비정상 반복 생성 0)
- **TC → 자동화 코드 생성**: 고도화 진행 중. 현재 문법 유효율 79% 수준이며, 학습 데이터 확충과 사람-검수 루프로 품질을 끌어올리는 단계
- **테스트 실행(CI)**: Slack 슬래시 커맨드로 트리거 → self-hosted 러너에서 Appium 실행 → 결과 Zephyr 동기화 + Slack 회신 (실사용)
- **확장 방향**: 두 트랙을 연결해 **요구사항(기획서)부터 자동화 테스트 코드까지** 한 번에 생성하는 end-to-end 자동화로 확장 예정

## 저장소 구조

```
.
├── finetune/        기획서 PDF → TC 생성 파이프라인
├── code_gen/        TC → Appium 테스트 코드 생성 파이프라인
├── examples/        익명 합성 예시 (TC → 로케이터 컨텍스트 → 생성 코드)
├── ci/              Slack → GitHub Actions 테스트 실행 자동화
├── jira_client.py   Jira / Zephyr 연동
├── gui.py           Gradio GUI
└── requirements.txt
```

## 참고

- 익명화 버전이므로 학습 데이터·기획서·실제 로케이터·모델 가중치는 포함하지 않습니다.
- 사내 식별자(Jira 호스트, 프로젝트 키 등)는 placeholder로 치환했습니다.
- 코드는 구조와 구현 방식을 보여주기 위한 것으로, 실행에는 별도의 데이터·모델·환경 설정이 필요합니다.
